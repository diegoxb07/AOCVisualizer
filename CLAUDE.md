# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A client-only web app ("Mission Visualizer") for NOAA Aircraft Operations Center (AOC) hurricane-hunter flight data. It plays back flight-level instrument data and synced cockpit/radar (MMR) video, with map tracking, charts, a PFD, satellite overlay, and KML/clip export. Built for NOAA AOC Science Branch.

There is no build system, package.json, or test suite. The app was originally one ~3400-line `index.html`; it has since been split (for resilience - a syntax or top-level error in one part no longer takes down the whole page) into:
- [index.html](index.html) - markup only, plus the CDN `<script>` tags and the ordered list of local script tags.
- [css/app.css](css/app.css) - the former inline `<style>` block.
- `js/01-state.js` … `js/19-bootstrap.js` - the former single `<script>`, sliced **in original top-to-bottom order** into numbered files by subsystem.

The JS files are **classic (non-module) scripts loaded in numeric order**, so they all share one global scope exactly as before (top-level `let`/`const` are shared across the files; inline `onclick`/`oninput` handlers still resolve). Order matters - `js/07-ui-controls.js` declares the shared DOM refs (`canvas`, `ctx`, `video`, `trackerModeSelect`, …) that later files use, so do not reorder the tags or convert to ES modules (modules would break both the shared globals and `file://` loading). When editing, find code by subsystem (see the per-file banner comments) rather than by absolute line number - the old line numbers in this doc are approximate and now relative to each file. All libraries are still loaded from CDNs in the `<head>`: Tailwind (via CDN script, configured inline), Chart.js + chartjs-plugin-zoom, Three.js r128 + OrbitControls, netcdfjs, and Tesseract.js (OCR).

## Running / deploying

- There's no dev server or build step. Open [index.html](index.html) directly in a browser, or serve the directory with any static file server.
- Deployment is GitHub Pages via [.github/workflows/static.yml](.github/workflows/static.yml) (uploads the whole repo as-is) - [.github/workflows/jekyll-gh-pages.yml](.github/workflows/jekyll-gh-pages.yml) also exists but is the unused sample workflow. Either pushes to `main` trigger an upload, no compile step.
- There is no lint/test command. Verify changes by opening the page in a browser and exercising the upload/playback flow (see `/run` or `/verify` skill).

## Architecture (spread across the `js/*.js` files, sharing one global scope)

**Data model**: Uploading a `.txt` (tab-separated AOC flight-level log) or `.nc` (NetCDF) file runs it through `parseEntireFile()` (~line 2191), which normalizes rows into `allParsedData` - one object per timestamp with fields like `lat`, `lon`, `pAlt`, `gpsAlt`, `tas`, `windSpd`, `pitch`, `roll`, etc. `.nc` files are first converted to the same tab-separated text format (via `netcdfjs`) before being handed to the same parser. The known/plottable fields are declared centrally in `METRIC_DEFS` (~line 1006) - label, color, and chart y-axis per metric; `availableMetrics` is derived from whichever of those keys are actually present in the uploaded file.

Time filtering (`applyFiltersAndInit`) derives `filteredData` from `allParsedData` based on the start/end time inputs; almost everything downstream (map, charts, PFD, HUD) renders off `filteredData[currentIdx]`, never `allParsedData` directly.

**Playback engine**: `masterSyncEngineTick()` is the single requestAnimationFrame loop driving playback. If no video is loaded, it advances `currentIdx` through `filteredData` using a `playbackAccumulator` scaled by the selected speed. If a video is loaded, it instead drives the video's `currentTime`/`playbackRate` and calls `syncTelemetryToVideoClock()` to keep `currentIdx` following the video clock.

**Video sync (two modes, `videoSyncMode`)**:
- *Manual*: user types an MMR start time; `filteredData` index follows a fixed offset from the video clock.
- *Auto-Sync (OCR)*: `Tesseract.js` (`initOCR`/`ocrWorker`) reads the timestamp burned into the video frame (`performImmediateOcrLock` for the "Sync Now" button, and an ongoing check inside `syncTelemetryToVideoClock`) and derives `videoStartSeconds` from it. OCR results are sanity-checked against the loaded flight data's time range before being trusted. While Auto-Sync is hunting/realigning, a non-blocking `#syncingIndicator` ("Syncing…") pill is shown; its visibility is driven by `refreshSyncingIndicator()` (in `js/06-ocr.js`) off `isOcrRunning || forceOcrSyncNextTick`, called from the OCR/sync code paths. It is deliberately *not* a screen-locking overlay like `#loadingOverlay`.

**HiDPI**: the 2D map canvas is rendered devicePixelRatio-aware. `resizeCanvasLayout` sizes the backing store to `css × DPR` (CSS display size stays via `width/height:100%`); all projection (`getX/getY`), mouse mapping (`screenToGeo`), and culling math work in **logical CSS pixels** (`cssW`/`cssH`, set in `js/01-state.js`), and the renderers apply `ctx.setTransform(DPR,…)` as the base transform (screen-space UI like the measure stat-boxes / on-canvas buttons reset to `setTransform(DPR,…)`, not identity). When touching the renderer, keep coordinates logical and never reintroduce raw `canvas.width` into drawing/projection (it's the device-px backing size). The PFD canvas is intentionally still 1× (not DPR'd).

**Map rendering**: Two interchangeable tracker modes (`trackerModeSelect`):
- *2D* (`renderMapEngineFrame` + helpers `getX`/`getY`/`renderBackground` in `js/15-map-render.js`): hand-rolled 2D canvas projection (no map library) drawing coastline/state polygons (`mapFeatures`, fetched once from Natural Earth + US states GeoJSON on GitHub in `js/19-bootstrap.js`), the flight track colored by wind speed or temperature (`getPathColorRGB`/`getSpdColorRGB`), wind barbs, custom markers, and the measurement tool (polygon/circle/rectangle hit-testing in `js/04-geo-measure.js`). The **whole world** is drawn - features are culled by the current viewport (`isBoxInView`/`getVisibleGeoBounds`), not by the flight bounds, so zooming out (wheel min scale 0.06) reveals all continents for synoptic analysis. The polygon measure path is built and stroked in one pass *before* its vertex dots (drawing a dot mid-path with `beginPath()` used to wipe a 2-point line).
- *3D* (`init3D`/`build3DScene`/`update3DFrame`, ~line 1346+): a Three.js scene with the same GeoJSON extruded as flat polygons, a plane model, and a colored 3D flight path. `get3DCoord()` is the lon/lat/alt → Three.js-space projection shared by both the terrain and the track.

Both modes overlay an optional satellite image (`fetchSatelliteImage`, `GIBS_LAYERS` table - all in `js/02-satellite.js`), placed by `satImageBox`. `GIBS_LAYERS` has two kinds of entry: **polar swaths** (MODIS/VIIRS, keyed by a calendar day via the day-stepper, with overpass times looked up via NASA CMR `lookupModisGranuleTime`, drawn from NASA GIBS WMS EPSG:4326 GetMap via `fetchGibsWMS`) and two **archive GOES** entries (`isReconApi:true`: value `GOES-RECON` / "GOES-East (Archive)", `satellite:'goes-east'`, `subLon:-75`; and value `GOES-RECON-WEST` / "GOES-West (Archive)", `satellite:'goes-west'`, `subLon:-137`). Their `bands` list is a hardcoded 3-entry fallback (Clean IR band 13, IR-enhanced `ir4`, Water Vapor band 9) that `loadSatelliteProducts()` replaces at startup with the **live product list from `GET /v1/satellite/products`** - a discovery endpoint returning every band (currently 5/7/9/13, each with a `default_cmap`) and composite product (`sandwich`, `geocolor` - full-disk only, no bbox) the API can render, plus each spacecraft's active date range (used to refine `minDate`). Don't hardcode new bands/products here - that endpoint is the source of truth, and picks up server-side additions with no client change. Each product object carries `bboxSupported` (`false` only for the two composites); `band`+`cmap` are used for spectral bands, `product` for composites (`fetchReconApiTile`/`fetchReconApiSat` send whichever applies - the API ignores `band`/`cmap` when `product` is given). **`#satBandSelect` (the product picker) always starts on a blank "Choose a product…" placeholder for archive-GOES layers** (`updateBandOptions()`) - `fetchSatelliteImage`/`maybeAutoPrecacheSatellite` both no-op on an empty selection, so nothing fetches or caches until the user explicitly picks one; don't reintroduce an auto-selected default band, that was a deliberate reversal (previously it silently picked bands[0] and started fetching/caching immediately). **Archive GOES comes from the noaa-recon-api** (`RECON_API_BASE` = `https://joshmurdock.net/api`, CORS-open, no key - github.com/jjmurdock19/noaa-recon-api), which renders GOES ABI tiles **server-side from NOAA's S3 archive** (the API auto-resolves the spacecraft from the date - East: GOES-16/19, West: GOES-17/18), so it works for the *historical* dates these recon flights fall on - solving the old limitation that NASA GIBS only keeps a rolling ~90 days of GOES (this is the TC-ATLAS server-side approach, just hosted externally). That request is an **async job**: `fetchReconApiSat` builds a bbox tile request (`center=lat,lon` + `dims` in km, ~11× faster than full-disk) for the playback-clock 10-min bucket (`goesBucketMs()` rounds flight UTC down to `cadenceMin`, `goesTimeStr()` formats it - so imagery advances as the tracker crosses each 10-min mark and the `~HH:MMZ` dropdown/badge label ticks with it) - UNLESS the picked product's `bboxSupported` is `false` (a composite), in which case bbox is skipped entirely and it renders the whole disk (slower, no upfront full-timeframe build - see below) - and `fetchReconApiTile` GETs `/v1/satellite/tile`, polls `/v1/satellite/status/{key}` every 3s until `ready` (≤30s cap), then loads the returned PNG into a canvas (`loadImageToCanvas`) and places it by the API's returned `bounds` (`[[lat_s,lon_w],[lat_n,lon_e]]` → `satImageBox`). Like the polar path, `fetchReconApiSat` marks `lastSatFetchTime` **synchronously** before debouncing (otherwise per-frame playback calls keep resetting the debounce timer and a new bucket never loads) and **stale-guards** the async result (`lastSatFetchTime !== fetchId`) against a mid-poll layer/band change. `subLon` + `goesInCoverage()`/`flightToSubSatAngle()` disable (grey out) each GOES option when the flight is outside that satellite's Earth-disk view (>~65° geocentric) - so an Atlantic flight greys out GOES-West and an east-Pacific flight greys out GOES-East. `computeSatFetchBox` (a wider synoptic box) is still used by the MODIS/VIIRS path only. The `#satLoadingOverlay` is a small non-blocking status pill (shown while a tile fetches/renders, not a full-screen cover), and the `⏪10 min / ⏩10 min` buttons (`#satStepCluster`, overlaid on the 2D tracker top-center; `skipFlightMinutes`) step the playhead - and thus the GOES scan - by 10 flight-minutes. The band picker (`#satBandSelect`) is a real dropdown - note it must be shown via inline `style.display` (it intentionally has no Tailwind `hidden` class, since `style.display=''` can't override a class). MODIS/VIIRS still work for any date back to their mission start.

**Auto full-timeframe pre-cache**: `maybeAutoPrecacheSatellite()` auto-builds an archive-GOES layer's imagery for the ENTIRE loaded flight up front (no confirm() prompt - it used to ask, don't reintroduce that), so scrubbing/playback never waits on the API mid-flight. It only runs once the user has **explicitly picked a product** from `#satBandSelect`'s placeholder (see above) - no "cache every band" fallback - and only for `bboxSupported` products; a full-disk-only composite is skipped here and just streams per-frame like a polar layer instead (see `batchTargetsForFlight`'s `bboxSupported === false` guard). It's called from `updateSatelliteOptions()` (so re-loading a flight while a GOES-archive product is already selected rebuilds automatically) and from the `satelliteSelect`/`satBandSelect` change handlers (`js/07-ui-controls.js`). Before doing anything visible it **rechecks the actual cold cache** (`batchTargetsForFlight(...).every(t => satBlobStore.has(t.fetchId))`), not a one-shot session flag - if every tile this flight/layer/product needs is already local it stays completely silent (no toast, no progress pill, no network requests); this recompute-fresh-every-time approach also means a *cancelled* or partial pass gets retried automatically next call, rather than being permanently skipped. `batchCaching` guards against piling on a second concurrent pass. When there IS missing work it delegates to the existing `precacheCurrentFlight()`, whose loop already `await`s each tile's full async job (request → poll `/status` → `ready`) before advancing `done`/`total` - so `#satPrefetchBar`'s percentage reflects tiles actually rendered and loaded, not merely requested, and only reports done once the loop truly finishes (or is cancelled via `#satPrefetchCancel`). The multi-flight "Batch Cache Satellites" modal (`populateBatchBandChecks`) filters the same way - composites never appear as cacheable checkboxes there either.

> **Note:** the noaa-recon-api is actively developed, so treat `loadSatelliteProducts()`'s live result as authoritative over anything written here - this doc's band/product list (5/7/9/13, `sandwich`, `geocolor`) reflects what existed when last checked, not a guarantee of what's current. Check `GET /v1/satellite/products` (or github.com/jjmurdock19/noaa-recon-api and `API.md`) before concluding a product isn't available - new ones need NO client-side change, they just start appearing in the dropdown.

**NOAA Recon Archive browser** (`js/12b-recon-archive.js`): a "one-stop-shop" alternative to the manual file upload - cascading Year → Storm → Mission dropdowns (`#reconYearSelect`/`#reconStormSelect`/`#reconMissionSelect`), backed by the noaa-recon-api's `/v1/recon/*` and `/v1/storms/*` endpoints (same `RECON_API_BASE` as the satellite overlay, defined in `js/02-satellite.js`). Clicking "Load Flight + Storm Track" (`loadReconMission`) fetches `/v1/recon/mission/{mission_id}` for metadata (storm/aircraft/year/`source_url`), then its **primary path** streams the mission's original **full-resolution** NetCDF straight through `GET /v1/recon/mission/{mission_id}/download` (`fetchArrayBufferWithProgress()`, reporting real byte progress via `Content-Length` + a `ReadableStream` reader into `#loadingOverlaySubtext` and `#reconArchiveStatus`) and runs it through the **same** `ncArrayBufferToTsv()` + `parseEntireFile()` pipeline a manual `.nc` upload uses - so every recorded variable (attitude, radar, thermodynamics, etc.) is available, not just the ~7-field preview. This endpoint used to be a redirect to NOAA's non-CORS archive (unreachable by `fetch()` from a browser); it's now a direct CORS-open stream through the API itself, which is what makes this automatic. **Fallback**: if the full download or parse fails (slow link, malformed file, etc.), it falls back to the `/v1/recon/mission/{mission_id}` JSON's **0.2 Hz-decimated** `[unix_time, lat, lon, wind_kt, wind_dir, sfmr_kt, alt_m]` track via `reconObsToTsv()` (same tab-separated repacking trick, padded to `parseEntireFile()`'s 10-column minimum) - `isNcFile`/status text reflect which path actually ran. The `⬇ .nc` link (`mission.source_url`) still opens the same original file directly for use in other tools - it's no longer the only way to get full resolution, just a convenience. `loadStormTrackForMission()` then fetches `/v1/storms/{year}/{name}?basin=` (basin = the first two characters of the mission's ATCF `storm_id`, e.g. `AL142024` → `AL`) for the storm's **whole-life** best-track (independent of the flight's own time window) into `stormTrackPoints`, drawn as a dashed, intensity-colored (`stormWindColor()`) polyline in both `drawStormTrack2D()` (`js/15-map-render.js`, called from `renderMapEngineFrame` before the flight track) and `build3DScene()` (`js/07-ui-controls.js`, alt pinned to 0 via `get3DCoord`) - `getX`/`getY`/`get3DCoord` project it exactly like any other layer since they're linear in lon/lat, not scoped to the flight's own bounds. The `#stormTrackToggleLabel` checkbox (map controls) and `#stormStatusCard` (nearest best-track fix to the current playback time - refreshed alongside the sat badge from `updateVisualComponents()` in `js/17-charts.js`) only appear once a track is loaded; `#stormStatusCard` lives **inside the tracker pane**, stacked below `#satTimeBadge` in the shared `#mapInfoStack` flex column (NOT the left control column - it was moved there and back once already, don't re-relocate without being asked). `parseEntireFile()` resets `stormTrackPoints`/`reconArchiveMeta` at its own top (not inside `applyFiltersAndInit`, which also runs on every start/end-time filter change and must NOT wipe the track) so a manual upload after an archive load doesn't keep showing a stale storm's track.

**HUD/PFD overlays**: `renderHUD()` and `renderPFD()` (~line 2733/2817) draw the text telemetry box and the attitude indicator respectively, reading from the currently interpolated row.

**Smooth interpolation**: `getInterpolatedRow()` (~line 1163) does Catmull-Rom cubic interpolation (with circular-angle unwrapping for heading/track) between the surrounding `filteredData` samples, used when "8Hz Smoothing" is enabled so playback isn't locked to the native sample rate of the log.

**Charts**: Chart.js instances are built in `buildChartLayout()` for the fixed set of canvases (`tempChart`, `navChart`, `attChart`, `altChart`, `tasChart`, `vertWindChart`, `sfcChart`, `thermoChart`) plus a free-form "Create Your Own Graph" (`masterChart`/`parameterChart`) where the user picks arbitrary metrics from `METRIC_DEFS` via `buildMasterMenu()`/`toggleMasterMetric()`. `updateVisualComponents(idx)` (~line 3031) is the per-frame fan-out that updates the map, charts, HUD/PFD, and timeline display together - this is the function to extend when adding a new synced visual element.

**Export**: KML export (flight path placemark) and a "Record Clip" feature that uses `navigator.mediaDevices.getDisplayMedia` + `MediaRecorder` to capture a screen-share of the tab while auto-playing a selected timeline range, saving a `.webm`.

## Working in this file

- Everything is global state (top-level `let`/`const` in the script - `allParsedData`, `filteredData`, `currentIdx`, `mapScale`, etc.) and DOM lookups by `getElementById`, not a component framework. Follow that style rather than introducing modules/build tooling.
- When adding a new flight-data metric, add it to `METRIC_DEFS` so it's auto-detected, plottable in "Create Your Own Graph", and unit-converted by `getConvertedVal`/`getMetricLabel`.
- When adding a new per-frame visual, wire it into `updateVisualComponents()` so it stays in sync with both manual playback and video-synced playback.
- Imperial/metric conversion and 2D/3D tracker parity are both intentional - when changing rendering logic, check whether the equivalent code path exists in both `renderMapEngineFrame` (2D) and `build3DScene`/`update3DFrame` (3D).
