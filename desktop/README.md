# Desktop launcher

Turns AOC Mission Visualizer + the noaa-recon-api into a single portable desktop app: no install,
no admin rights, opens in its own window instead of a browser tab.

## How it fits together

- **`app.py`** is the entry point. It checks GitHub for updates, provisions a Python runtime for
  the API on first run, starts the API as a subprocess and the frontend on a stdlib static server,
  then opens a native window (`pywebview`) pointed at the frontend.
- The API (`noaa-recon-api`, downloaded from GitHub - never vendored into this repo, see
  `updater.py`) runs **unmodified**, in its own provisioned Python environment
  (`provision.py`), not frozen into this app - `netCDF4`'s native extensions are painful to
  freeze with PyInstaller but install fine as an ordinary wheel.
- `js/02-satellite.js` deliberately treats the two features it depends on differently, because
  GOES image rendering is the compute-heavy part this whole app exists to take off the hosted
  API, while archive/storm-track lookups are lightweight:
  - **Satellite imagery** (`RECON_SATELLITE_API_BASE`) is served **only** by the local API when
    `window.__AOC_LOCAL_API__` is set (which this launcher injects) - it never falls back to the
    hosted API. If the local service is down, `satelliteApiHealthOk` goes false, the GOES band
    picker greys out with "(API Offline)", and the `#localApiHealthBadge` pill in the header shows
    "Local Imagery: Offline" - there is no automatic or manual hand-off to the hosted server for
    this.
  - **Archive/storm-track lookups** (`RECON_ARCHIVE`/`RECON_HOSTED_API_BASE`, in
    `js/12b-recon-archive.js`) always go through the hosted `joshmurdock.net` API, tracked by its
    own independent `archiveApiHealthOk`, regardless of whether the local satellite service is up.
  - The hosted website (GitHub Pages, no `window.__AOC_LOCAL_API__`) is unaffected either way -
    both bases fall back to hosted there, exactly as before.

## Running for development

```
cd desktop
pip install -r requirements.txt
python app.py
```

This serves the **repo root directly** as the frontend (no snapshot copy), so editing
`index.html`/`js/*.js`/`css/*.css` and relaunching picks up changes immediately. Downloaded/
generated state (the vendored API source, its provisioned runtime, `state.json`) lives under
`desktop/.devdata/` so it never touches the actual repo.

## Building the portable app

```
cd desktop
pip install -r requirements.txt pyinstaller
python build.py
```

Produces `desktop/dist/AOCVisualizer/` - copy that whole folder anywhere (USB stick, shared
drive, another machine) and run `AOCVisualizer.exe`. First launch on a machine downloads the API
source and provisions its Python runtime (needs internet, takes a couple of minutes); after that
it's offline-capable for everything except archive loading and GOES imagery (see
`docs/CONNECTIVITY.md`).

## Icon

`icon.ico` (generated by `build_icon.py` - a radar-sweep-and-aircraft glyph in the app's own
navy/cyan palette, no external asset needed) is embedded into the exe itself
(`AOCVisualizer.spec`'s `EXE(icon=...)`) and also bundled as a data file so `app.py` can pass it
to `webview.start(icon=...)` for the window/taskbar icon at runtime. Regenerate after a design
change with `python build_icon.py`, then rebuild.

## Known limitations (v1)

- **Windows only** for now. The embeddable-Python provisioning step (`provision.py`) is a
  Windows-specific artifact; a Linux port would likely shell out to a system `python3 -m venv`
  instead.
- The **self-updater keeps `frontend/` and the vendored API current**, but does not replace the
  launcher's own frozen exe code while running - safely swapping a running Windows executable
  needs a separate restart-and-swap mechanism. Launcher-code changes (rare - it's thin glue) ship
  as a new build that gets manually re-copied to office machines.
- Requires the **WebView2 Runtime** (bundled with modern Windows/Edge; present on virtually all
  current Windows 10/11 machines).
